Choosing a format

Reach for SD-JWT VC when…

  • You target the EUDI Wallet, OID4VCI, or OID4VP.
  • You need compact, selectively disclosable credentials out of the box.
  • You want predicate-style minimal disclosure (e.g. age_over_18).

Reach for W3C VC 2.0 when…

  • You operate in a linked-data / JSON-LD ecosystem.
  • You need semantic interop and rich @context vocabularies.
  • You rely on Data Integrity proofs or existing VC 2.0 tooling.

Both come from the same JSON Schema claim set, so you can issue either (or both) without redefining your data. See the registry for each type.

Issue an SD-JWT VC

Split the claim set into always-visible claims and selectively disclosable Disclosures; the payload carries only digests in _sd (RFC 9901).

Present only what's needed

The holder keeps the issuer-signed payload and sends only the Disclosures they choose. Withheld claims remain digests the verifier cannot reverse.

Verify a presentation

Each presented Disclosure must hash to a digest in _sd. (A production verifier also checks the issuer signature, vct, validity, and key binding.)

Issue a W3C VC 2.0 (JSON-LD)

The same claims wrapped in the credentials/v2 base context plus the type's Glide context.
